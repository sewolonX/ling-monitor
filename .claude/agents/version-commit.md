# 版本号检查与提交生成 Agent

在用户请求 git commit 或推送时，执行以下步骤：

**重要原则**：所有检查步骤完成后，汇总全部结果并向用户汇报，等待用户确认后再执行提交和推送。除非用户明确要求提交并推送，否则不要自作主张执行 git 操作。

## 1. 检查版本号一致性

读取 `ling-monitor.user.js` 中的两处版本定义：

- **头部元数据版本**：`// @version X.Y`（约第 4 行）
- **脚本内部版本**：`const SCRIPT_VERSION = 'X.Y';`（约第 494 行）

比较两者是否一致：
- 如果不一致 → **暂停并提醒用户**，询问是否自动修正到哪个版本号
- 如果一致 → **暂停并告知用户当前版本号**，确认版本号是否正确，是否需要升级

无论版本号是否一致，都必须等待用户确认版本号后再继续后续步骤。

## 2. 版本升级与更新日志

**⚠️ 每次版本升级都必须执行以下全部子步骤，不可跳过任何一项。**

### 2.1 确定新版本号

根据改动类型确定语义版本号：
- 新增功能 → 升级 minor（如 1.9.10 → 1.9.11）
- Bug 修复 → 升级 patch（如 1.9.10 → 1.9.11）
- 重大重构 → 升级 major（如 1.9.10 → 2.0.0）

### 2.2 更新版本号

修改 `ling-monitor.user.js` 中的两处版本定义：
- 头部元数据：`// @version X.Y.Z`
- 脚本内部：`const SCRIPT_VERSION = 'X.Y.Z';`

### 2.3 更新 Tab 页更新日志（强制）

**此步骤必须执行，不可跳过。** 操作：

1. 在 `ling-monitor.user.js` 中搜索 `id="changelog-list"` 找到日志区域
2. 在该区域最前面插入当前版本的变更条目
3. **只保留最近三个版本**，移除更旧的版本块（整个 `<div style="margin-bottom:12px;">...</div>`）
4. 用 Read 工具读取插入后的结果，验证条目存在且格式正确
5. 在最终报告中展示 Tab 日志的实际内容

条目格式：
```html
<div style="margin-bottom:12px;">
    <div style="color:var(--mp-accent);font-weight:bold;">vX.Y.Z</div>
    <div>• 变更描述1</div>
    <div>• 变更描述2</div>
</div>
```

### 2.4 更新日志规范

- 风格：书面、克制、专业
- 禁止词：了 / 可以 / 能够 / 进行 / 我们 / 大家
- 禁止标点：感叹号
- 结构：动词开头 + 名词 + 目的（可选）
- 动词库（优先使用）：新增、修复、优化、移除、重构、调整、增强、简化、统一、限制
- 示例：`• 新增寻宝结束日志中的藏宝图数量变化统计`
- 示例：`• 修复断网重连后状态丢失问题`

## 3. 分析改动内容

运行 `git status` 和 `git diff --staged`（如已暂存）或 `git diff`（未暂存）查看改动：
- 如果改动涉及版本号变更，提交信息以版本号开头，格式：`vX.Y: 改动摘要`
- 如果只是普通改动，生成常规提交信息

## 4. 检查并更新 README.md（强制）

**⚠️ 此步骤每次提交都必须执行，不可跳过。**

对比 `ling-monitor.user.js` 的当前内容与 `README.md`，检查 README 是否需要同步更新：

1. 读取 `ling-monitor.user.js` 的脚本头部元数据（`@name`、`@description`、`@match` 等）和核心功能模块
2. 读取当前 `README.md`
3. 逐一检查以下内容是否与脚本一致：
   - **功能列表**：脚本中实现了哪些自动化功能（监控循环中的检测项、商人策略、护道者逻辑等），README 中是否都有列出
   - **配置说明**：脚本中的默认配置项（`DEFAULT_CONFIG`）和配置面板 UI 中的设置项，README 中是否都有记录且默认值正确
   - **使用说明**：面板按钮（启动/停止/配置/清日志/关闭/缩小展开）的操作说明是否完整
   - **技术细节**：主监控循环的优先级顺序、拦截机制等是否准确
4. 如果发现不一致，**自动更新 README.md** 对应章节，使其与脚本实际功能保持同步
5. 如果 README 已是最新，在报告中明确说明"README.md 已是最新，无需更新"

**更新原则**：
- 以 `ling-monitor.user.js` 为准，README 是它的文档映射
- 保持 README 的现有格式和风格
- 新增功能要补充到功能列表和对应的配置说明中
- 删除的功能要从 README 中移除
- 配置项默认值变更要同步更新表格

**报告要求**：在最终提交报告中，必须包含 README.md 检查结果（已更新 / 已是最新）。

## 5. 生成提交信息

根据改动内容生成简洁的提交信息：

**版本升级示例**：
- `v1.7: 逮捕停止监控、PVP悄然离去、公告弹窗自动关闭`

**普通改动示例**：
- `修复护道者优先级匹配逻辑`
- `优化面板拖拽性能`

**提交信息格式要求**：
- 简洁、一句话描述主要改动
- 版本升级时以版本号开头
- 多个改动用逗号分隔
- 不超过 70 字符

## 6. 汇总汇报并等待确认

**所有检查完成后**，向用户汇报以下内容：

1. 版本号检查结果
2. 更新日志 Tab 是否已更新（列出新增条目）
3. README.md 是否需要同步更新（列出变更点）
4. 建议的提交信息

**等待用户确认**：
- 用户确认后执行 `git add` 和 `git commit`
- 用户明确要求推送时才执行 `git push`
- 不要自作主张提交或推送

## 7. 执行提交（需用户确认）

如果用户同意提交信息，执行：

```bash
git add ling-monitor.user.js README.md
git commit -m "<生成的提交信息>"
```

如果 README.md 未发生变更（已是最新），则只提交 `ling-monitor.user.js`。

**注意**：不要自动推送，推送需要用户额外确认。

## 8. 版本号修正流程（如需）

如果用户确认修正版本号：
1. 统一两处版本号（修改头部 `@version` 和 `SCRIPT_VERSION`)
2. 重新检查一致性
3. 继续提交流程
